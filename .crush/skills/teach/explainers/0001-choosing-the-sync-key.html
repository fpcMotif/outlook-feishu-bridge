Customer Mirror · Learning record 0001

Which field do you key the mirror on?

Best practice for the unique key that keeps the Convex customers table in lockstep with the Feishu Bitable customer-info table — and never drifts to 3× again.

The one-line answer: key the mirror on the source system's own immutable id — the Feishu API record_id — never on a human-editable / formula column like the Record Id field. Then pair that key with a reconcile step (the prune) so deletes in the source propagate. Key alone is not enough.

1 · The principle: stable identity vs. business meaning

Every row needs an identity to upsert on. You have two kinds of candidate:

KindWhat it isGood sync key?
Natural keyA value with business meaning — email, account no., the Record Id column.risky meaning changes
Surrogate keyA system-assigned id with no meaning outside the system — Feishu's API record_id.yes immutable by design

The trap is that "immutable" business identifiers usually aren't. The classic cautionary tale: a startup keyed everything on the user's email — then aliases arrived and they couldn't change an email without breaking every linked row. Apply the "never changes" test skeptically: "never" has a way of becoming "well, once."

2 · Your exact case

The Bitable customer table exposes two things that look like an id:

CandidateSourceStable?
item.record_id
e.g. recvl0iItb47Dh		Assigned by Feishu. The system of record's own surrogate key.immutable never changes while the row exists
fields["Record Id"] columnA RECORD_ID() formula column — equals the API id today, but a human could switch it to a manual field.mutable a "smart" value with business meaning

Your old code keyed on the column:

// before — keyed on a mutable formula column
recordId: recordIdFromCustomerInfoRow(item)   // fields["Record Id"] ?? item.record_id

// after (ADR-0021) — keyed on the immutable surrogate
recordId: item.record_id

Why this caused 3×: when source rows are deleted & re-imported, Feishu mints fresh record_ids. An upsert-only mirror inserts the new generation but can't see that the old keys are gone — so generations stack up. (The column choice is a second, latent re-key trigger on top of that.)

3 · The "never changes" test — how to pick a key for any source

1
Does the source system assign its own id that is stable for the life of the row? → use it. (record_id, a Git SHA, a Stripe cus_….)
2
Is your candidate something a human can edit, a formula, or derived from business data (email, name, code)? → do not key on it. Keep it as a display/alternate field instead.
3
Could the answer to "does it ever change?" be "once in a blue moon"? → treat that as yes, it changes. Pick the surrogate.

4 · Key alone isn't enough: the upsert + reconcile pair

A stable key gives you idempotent upsert — re-running the sync can't duplicate rows. But upsert never deletes. To stay in lockstep you need the second half: reconcile (your Mirror Prune).

  • Upsert each source row by record_id (insert if new, patch if changed) — safe to retry.
  • Reconcile: after a complete pass, delete any mirror row whose record_id wasn't seen this run. Those are orphans from source deletes/re-imports.
  • Gate the delete on a verified-complete sync, so a half-finished fetch can never wipe live rows.

Convex gotcha — don't reconcile by timestamp

A Convex query is re-run when its data changes, not when the clock ticks — so a "mirroredAt older than X" staleness check is unreliable. Track liveness as a set of ids seen this run (or a generation number) instead. That's exactly why your prune builds an in-run seenRecordIds set rather than comparing timestamps.
