LR-0001 · Outlook taskpane uploads

Why the icon fill restarted near 30%

Not a wrong Convex “route” — a race on the normal path: two async starts for the same file id, then state/UI accepted a lower progress than XHR had already reported.

One line: Register the in-flight Promise synchronously, merge progress with Math.max, show indeterminate fill only while pending, and smooth display with a monotonic target.

Sequence: duplicate start (bad) vs dedupe (good)

1 · In-flight dedupe (orchestration)

BAD Late inFlight.set

Gap after check lets two uploads start; second dispatch can pass progress: 0.

export async function uploadBad(id, dispatch) {
  if (inFlight.has(id)) return inFlight.get(id)!;
  await generateUploadUrl(); // gap — TOCTOU
  const p = runXHR(id, dispatch);
  inFlight.set(id, p);
  return p;
}

GOOD Sync promise registration

From uploadIntakeFile.ts — check, create IIFE promise, set before caller awaits.

export function uploadIntakeFileToStorage(...) {
  const existing = inFlight.get(upload.id);
  if (existing) return existing;

  const tracked = (async () => {
    dispatch({ type: "uploadStatusChanged",
      id: upload.id, status: "uploading" });
    // ... XHR onProgress → uploadProgressUpdated
  })().finally(() => inFlight.delete(upload.id));

  inFlight.set(upload.id, tracked);
  return tracked;
}

2 · void + non-async starter (routing)

BAD async wrapper re-enters

export async function queueBad(uploads, dispatch) {
  for (const u of uploads) {
    await uploadBad(u.id, dispatch);
  }
}

Serializes picks; still racy if uploadBad awaits before set.

GOOD Fire-and-forget same promise

export function queueIntakeFileUploads(...) {
  for (const upload of uploads) {
    if (inFlight.has(upload.id)) continue;
    void uploadIntakeFileToStorage(deps, upload, dispatch);
  }
}

Second queue pass hits inFlight.has — test: one XHR call.

3 · Reducer: monotonic progress

BAD Blind assign

case "uploadProgressUpdated":
  return map(file =>
    file.id === action.id
      ? { ...file, progress: action.progress }
      : file);

30 then 5 → bar jumps backward.

GOOD Math.max

case "uploadProgressUpdated":
  progress: Math.max(file.progress ?? 0, action.progress),

case "uploadStatusChanged":
  progress: action.status === "uploading"
    ? Math.max(file.progress ?? 0, action.progress)
    : action.progress,

Re-enter uploading with 0 keeps 30 — see intakeReducer.test.ts.

4 · Row UI: indeterminate only while pending

BAD Indeterminate for whole upload

// AttachmentSectionRows (anti-pattern)
const uploadIndeterminate = uploadActive;

XHR hits 30% but fill stays pulsing — then switches → feels like restart.

GOOD Pending pulse, then determinate

const uploadActive =
  status === "pending" || status === "uploading" || ...;
const uploadIndeterminate = status === "pending";

<FileTypeIconWithUploadProgress
  progress={progress ?? 0}
  active={uploadActive}
  indeterminate={uploadIndeterminate}
/>

5 · Display layer: ignore spurious zero

BAD Reset display on any 0

// useSmoothedUploadProgress (anti-pattern)
if (progress <= 0) {
  displayRef.current = 0;
  setDisplay(0);
}

Reducer still 30, prop flickers 0 → fill restarts.

GOOD Monotonic target + ease

// uploadDisplayProgress.ts
export function uploadDisplayProgressTarget(prev, incoming, active) {
  if (!active) return incoming;
  if (incoming <= 0 && prev > 0) return prev;
  return Math.max(prev, clamped(incoming));
}

Regression tests to remember
